# skregexp (Fork for NanaTerry)

## 概要
このリポジトリは、Delphi用の正規表現ライブラリ [skregexp](https://github.com/shukomiya/skregexp) のフォークである。
オリジナルは長期間更新が停止しているため、テキストエディタ「NanaTerry」の開発継続に必要なバグ修正および機能維持を目的として、独自にメンテナンスを行っている。

## フォークの目的と位置づけ
本フォークは汎用的な利用を意図したものではなく、以下の特定の環境・目的のために運用されている。

*   **レガシー環境のサポート**: Delphi 2009 環境での動作を保証する（Delphi XE2以降の `System.RegularExpressions` が使用できない環境向け）。
*   **バグ修正**: オリジナル版に含まれる未修正の不具合への対応。
*   **AIペアプログラミング用**: コードベースの把握と修正履歴の明確化。

## 特徴
標準の正規表現ライブラリと比較し、本ライブラリには以下の利点があるため継続利用している。

*   **全角/半角の区別**: 日本語処理において重要な、全角文字と半角文字の厳密な区別が可能。
*   **軽量・高速**: Delphiネイティブ実装であり、外部DLLを必要としない。

## 免責事項
*   Delphi XE2 以降を使用している開発者には、標準ライブラリ `System.RegularExpressions` の使用を推奨する。
*   本リポジトリの変更内容は、NanaTerryの仕様に特化している可能性がある。

# 修正履歴（メモ）

## )...戻り読み内で可変長文字列は使えません; (?<=あ{3})あ <--
* 「(?<=戻り読み)」で可変長を使えないのはその通りだが
* 「{3}」は３回繰り返すという意味であり、可変長ではない。
* 「(?<=あ{3})あ」＝「(?<=あああ)あ」

* 同じく可変長を使えない「(?=先読み)」で「あ(?=あ{3})」をしても怒られないし、正規表現デバッグでは両方正常にヒットする。

#### 「戻り読み内の量指定子対応」`opBound` ブロックの修正


## エスケープし忘れなどで「^*」検索したらフリーズしていた
原因は **「ゼロ幅マッチ（文字を消費しないマッチ）に対する繰り返し処理」** の考慮漏れ。

#### 秀丸/Mery ではエラーとならずにキャレットが移動する
 **「ゼロ幅マッチの許容と強制進行」** という実装によるものと推測。
1. **許容**: 一部のエンジンは `^*` を「行頭に 0回以上マッチ」と解釈し、常に成功（ただし 0文字マッチ）とみなす。
2. **強制進行**: 0文字マッチを許容するエディタは、無限ループを避けるために「マッチした長さが 0 だった場合、次の検索開始位置を強制的に +1 する」というロジックを持っている。
   - `^` にマッチ（長さ0） → ヒット表示はできない（長さ0のため） → 次回検索のためにキャレットを 1 つ右へ移動。
   - これにより、ユーザーからは「何も起きないがキャレットだけ動く」ように見える。

#### 検索エンジンとしての整合性
モダンな正規表現エンジン（PCRE, .NET, ICU等）の多くは、`^*` を **「コンパイルエラー」** とする。これが最も開発者にとって親切な挙動。
秀丸等の挙動は、古いエンジンとの互換性や、検索を中断させないための「寛容な設計」の結果と言えるが、プログラミング用途のエディタとしては、明確にエラーを出す方がバグの早期発見に繋がると思われるので適切なエラーを出すように修正。


## 「後ろ読み否定(?<!)」の中に「または|」があり、区切られた検索語句が１文字と２文字以上だと、１文字側の条件が無視される
「戻り読み」
本文

```pascal
	ABCDE
```

検索

```pascal
(?<!B|XY)C
(?<!XY|B)C
(?<!XY|B|123)C
```

文字数が違ったり、または|を使わず分割したり、後ろ読み肯定や先読み否定は正常

```pascal
(?<!A|B)C
(?<!B|X)C
(?<!AB|XY)C
(?<!AB|X)C
(?<!X|AB)C

(?<!B)(?<!XY)C

(?<=B|XY)C
(?<=XY|B)C
(?<=XY|B|123)C

C(?!D|E)
C(?!DE|XY)
C(?!D|XY)
```

#### 「戻り読み内の分岐（OR条件）」および「長さの異なる条件」が混在しても、正確に判定できるようになった。

##`[いずれか]` かつ `可変長` で `\K` を使ったとき、最終的な目的文字が[いずれか]に含まれているヒットしない

本文
```
ABCDEFG
```
問題
```
[ABC]+\KC
[ABC]*\KC
[ABC]{1,}\KC
[ABC]{1,3}\KC

[A-Z]+\KC
[A-Z]*\KC
[A-Z]{1,}\KC
[A-Z]{1,3}\KC
```
### 1. `nkKeepPattern` (`\K`) から `Stack.Clear` を削除
*   **以前の挙動:** `\K` を通過した瞬間、それまでのバックトラック履歴（やり直しのための記憶）を全て消去していた。
*   **修正後:** 履歴を消さずに、開始位置（`StartP`）だけを更新するようにした。

### 2. `nkPlus` / `nkStar` のループ条件を `<=` から `<` に変更
*   **以前の挙動:** `SubP <= AStr` だったため、「最長マッチした位置（`AStr`）」もバックトラック用のスタックに積んでいた。
*   **修正後:** `SubP < AStr` とし、最長位置はスタックに積まないようにした。
*   **安全性:** ループを抜けた直後、エンジンは必ず「最長マッチした位置（`AStr`）」で次の処理を実行する。もしスタックにも同じ位置を積んでしまうと、バックトラックが発生した際、**「全く同じ最長位置で、もう一度同じ処理をやり直す」** という無駄（重複試行）が発生する。これを `<` に修正したことで、重複試行が排除され、パフォーマンスと正確性が向上した。
### 3. `TREMatchEngine.Match` での `StartP` 上書きを削除
*   **以前の挙動:** `MatchEntry`（検索の本体）が終わった後、呼び出し元で強制的に `StartP` を「検索を開始した位置」に上書きしていた。
*   **修正後:** 上書き処理を削除した。
*   **安全性:** `MatchEntry` の一番最初で、すでに `StartP` は「検索を開始した位置」に初期化されている。そのため、呼び出し元で再度上書きする必要は全くなかった。この不要な上書きが `\K` の変更を握り潰していた元凶だ。削除しても、通常の検索（`\K`を使わない検索）では `MatchEntry` の最初でセットされた値がそのまま使われるだけなので、既存の動作は一切変わらない。

## (?<!後ろ読み否定) の再修正
1. 「(?<!後ろ読み否定)」の中に、最終的な目的文字が含まれていると読みこみ違反
2. 「(?<!後ろ読み否定)」の中に、最終的な目的文字が含まれていないとヒットしない

対象文字列: `  テスト 123 test `
**ケース 1: `(?<! ) ` （直前がスペースではないスペース）**
**ケース 2: `(?<![a-z ])2` （直前が英小文字でもスペースでもない '2'）**
**ケース 3: `(?<! ) ` （直前がスペースであるスペース：否定条件に引っかかるケース）**

「目的文字が含まれているとアクセス違反になる」問題も、「含まれていないとヒットしない」問題も、今回の修正（`MatchSpecial` に `nil` を渡す、ポインタ管理の厳格化）によって解消された。
